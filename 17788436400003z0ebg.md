## 1. Introduction

This project designs and implements a property graph database in Neo4j for an airline route dataset. The objective is to transform the tabular dataset into a graph model that can support route-based analysis, airport connectivity queries, airline competition analysis, and higher-level network exploration. The graph database designed using the Arrows App and implemented in Neo4j through an ETL process that generated separate CSV files for nodes and relationships.

The selected graph model represents airports and countries or regions as nodes, while flight services are modelled as directed ROUTE relationships between airport nodes. This design was chosen because the required queries rely heavily on airport to airport traversal, route aggregation, and variable-length path search. In particular, modelling ROUTE as a relationship allows the graph to naturally support questions such as finding multi-hop travel paths between Beijing Capital International Airport and Perth International Airport.

The ETL process included data profiling, missing value checking, data cleaning, airport-country consistency validation, and CSV transformation. During cleaning, suspicious airport-country inconsistencies were identified and corrected using an external airport reference file where reliable matches were available. Cases that could not be reliably matched to the reference data were handled using a documented majority-rule fallback. The final imported graph contains Airport and Country nodes, ROUTE relationships between airports, and LOCATED_IN relationships from airports to countries or regions.

## 2. Property Graph Design

The proposed graph follows the labelled property graph model, where domain entities are represented as nodes, connections between entities are represented as relationships, and both nodes and relationships can store key-value properties [1], [3]. The design is centred on the airline route network and is intended to support all required Cypher queries in the project. The main analytical focus of the dataset is airport-to-airport route connectivity, so the graph model prioritises route traversal, airport-pair aggregation, domestic/international classification, and aircraft-type analysis.

![gh](https://cdn.jsdelivr.net/gh/extraterrestrial-being/markdown_images@main/17786434040007amqj9.png)
Fig. 1. Property graph schema designed in Arrows App.
### 2.1 Node Labels

The Airport node represents each distinct airport in the dataset. Each airport is stored once, rather than being separated into departure and arrival airport nodes. This avoids duplicating the same real-world airport and allows route traversal queries to pass naturally through airport nodes. This follows the general graph modelling principle that nodes should represent the main entities or discrete objects in the domain [1].

The properties of the Airport and Country nodes are summarised in Table 1.

| Node Label | Property      | Data Type | Description                                                  |
| ---------- | ------------- | --------: | ------------------------------------------------------------ |
| Airport  | airportId   |   Integer | Generated unique identifier for each airport                 |
| Airport  | airportName |    String | Full name of the airport                                     |
| Airport  | city        |    String | City in which the airport is located                         |
| Airport  | countryName |    String | Country or region assigned to the airport after ETL cleaning |
| Country  | countryName |    String | Name of the country or region                                |
Table 1. Node labels and properties in the proposed graph schema.

The Country node represents each country or region associated with airports in the dataset. It is included as a separate node label because country or region information is used in domestic/international route classification and country-level analysis.


### 2.2 Relationship Types

Each airport is connected to its country or region through a directed LOCATED_IN relationship:

```cypher
(:Airport)-[:LOCATED_IN]->(:Country)
```

This relationship represents the geographical association between an airport and a country or region. It also allows country-level queries to use graph structure rather than relying only on text properties stored on airport nodes. Relationships in Neo4j connect a source node to a target node, have a relationship type, and may also store properties that describe the connection [1], [3].

The ROUTE relationship represents a directed airline service record from a departure airport to an arrival airport:

```cypher
(:Airport)-[:ROUTE]->(:Airport)
```

In the Arrows App design, ROUTE is shown as a directed self-relationship on the Airport label. This does not mean that an airport flies to itself. Instead, it means that one Airport node instance can be connected to another Airport node instance using the same Airport label. The direction of the relationship follows the direction of travel from the departure airport to the arrival airport.

The properties of the ROUTE and LOCATED_IN relationships are summarised in Table 2.

| Relationship Type | Direction                            | Property       | Data Type | Description                                              |
| ----------------- | ------------------------------------ | -------------- | --------: | -------------------------------------------------------- |
| LOCATED_IN        | (:Airport)-[:LOCATED_IN]->(:Country) | None           |         — | Connects each airport to its country or region           |
| ROUTE             | (:Airport)-[:ROUTE]->(:Airport)      | routeRecordId  |   Integer | Generated unique identifier for each route record        |
| ROUTE             | (:Airport)-[:ROUTE]->(:Airport)      | airlineName    |    String | Name of the airline operating the service                |
| ROUTE             | (:Airport)-[:ROUTE]->(:Airport)      | airlineCountry |    String | Country of the airline                                   |
| ROUTE             | (:Airport)-[:ROUTE]->(:Airport)      | planeNames     |    String | Aircraft type names stored as semicolon-separated values |
| ROUTE             | (:Airport)-[:ROUTE]->(:Airport)      | routeType      |    String | Derived value: domestic or international                 |
Table 2. Relationship types and properties in the proposed graph schema.

The routeType property is derived during ETL by comparing the cleaned country or region values of the departure and arrival airports. If both airports are located in the same country or region, the route is classified as domestic; otherwise, it is classified as international.

### 2.3 Design Justification

This model was selected because it directly supports the required project queries. Airline-related queries can filter ROUTE relationships by airlineName and airlineCountry. Domestic and international route counts can be calculated from the derived routeType property, and they can also be validated by comparing the countries connected to the departure and arrival airports. Airport-pair aggregation queries can group ROUTE relationships by their start and end airport nodes. Aircraft-type analysis can be performed by splitting and unwinding the semicolon-separated planeNames values. Graph modelling should be guided by the questions that the database needs to answer, because graph models and graph queries are closely related [1].

The most important reason for modelling ROUTE as an airport-to-airport relationship is that the project requires path traversal using at most three ROUTE relationships. This is naturally expressed in Cypher as a variable-length path pattern such as:

```cypher
(:Airport)-[:ROUTE*1..3]->(:Airport)
```

If ROUTE were modelled only as a separate node, this path query would become less direct because each flight segment would require traversal through an additional route node. Therefore, representing ROUTE as a relationship better matches the route-based and path-based requirements of the project.

The Country node is included to make country or region information part of the graph structure. Although country information could be stored only as an airport property, a separate Country node makes the model more extensible and supports country-level analysis, such as identifying the most frequent international country pairs. It also makes the schema easier to interpret because airport location is represented explicitly through the LOCATED_IN relationship.

### 2.4 Advantages and Limitations

**Advantages**

The main advantage of this design is that it keeps the graph structure simple while still supporting the required analytical tasks. Airport-to-airport traversal is direct, route records can be counted and grouped through ROUTE relationships, and country or region information is available through both airport properties and LOCATED_IN relationships. The model also avoids creating separate departure and arrival airport nodes, which would incorrectly split a single real-world airport into two entities and make transfer-path queries harder to express.

Another advantage is that the design stores airline and aircraft information on the ROUTE relationship, where it naturally describes a particular airline service record. This is appropriate for the required queries because airline names, airline countries, and aircraft types are mainly used for filtering, grouping, and counting route records.

**Limitations**

A limitation of this design is that Airline and AircraftType are not modelled as separate node labels. This means the graph is less suitable for deeper airline-centred or aircraft-centred traversal, such as analysing airline alliance networks or aircraft deployment networks. If those analyses were required, the model could be extended by adding Airline nodes and AircraftType nodes connected to ROUTE relationships or route-record entities. However, for the current project requirements, storing these values as ROUTE properties keeps the model clear, efficient, and aligned with the required Cypher queries. Graph database design has no single universal schema; the appropriate model depends on the intended query patterns and analytical use cases [1], [3].

To correct the suspicious airport-country cases, the ETL process used an external airport reference file, airports.csv[4], located in the raw_data folder. The correction process only targeted the 41 flagged suspicious airportName + city cases, rather than modifying the entire dataset. This limited the cleaning scope to cases that were actually detected as inconsistent during ETL.

The correction process used the reference airport file as the primary source where a reliable match was available. The reference matching considered airport name and municipality/city information. If the reference file provided a unique and consistent match, the country or region value was corrected using the reference result. ## 4. Neo4j Import and Graph Database Implementation

The graph-ready CSV files generated during the ETL process were imported into Neo4j using Cypher LOAD CSV WITH HEADERS. The import followed the property graph schema defined in Section 2:

```cypher
(:Airport)-[:LOCATED_IN]->(:Country)
(:Airport)-[:ROUTE]->(:Airport)
```

The implementation process consisted of five main steps: clearing the existing database, creating uniqueness constraints, importing Country nodes, importing Airport nodes, creating LOCATED_IN relationships, and creating directed ROUTE relationships.

### 4.1 Clearing the Existing Database

Before importing the new graph, all existing nodes and relationships were removed to ensure that the database started from a clean state. This prevented duplicate nodes or relationships from previous test imports.

```cypher
// Clear all existing nodes and relationships before import.
MATCH (n)
DETACH DELETE n;
```
### 4.2 Creating Constraints

Uniqueness constraints were created for Country.countryName and Airport.airportId. These constraints help prevent duplicate node creation and improve matching consistency when relationships are created.

```cypher
// Create a uniqueness constraint for Country nodes.
CREATE CONSTRAINT country_name_unique IF NOT EXISTS
FOR (c:Country)
REQUIRE c.countryName IS UNIQUE;

// Create a uniqueness constraint for Airport nodes.
CREATE CONSTRAINT airport_id_unique IF NOT EXISTS
FOR (a:Airport)
REQUIRE a.airportId IS UNIQUE;
```

![gh](https://cdn.jsdelivr.net/gh/extraterrestrial-being/markdown_images@main/1778606960000xyhxqm.png)
Fig. 16. Uniqueness constraints created for Country and Airport nodes.

### 4.3 Importing Country Nodes

The countries.csv file was used to create Country nodes. Each country or region appears once because the ETL process deduplicated the corrected country or region values before export.

```cypher
// Import Country nodes.
LOAD CSV WITH HEADERS FROM 'file:///countries.csv' AS row
MERGE (c:Country {countryName: row.countryName});
```

![gh](https://cdn.jsdelivr.net/gh/extraterrestrial-being/markdown_images@main/1778607009000jo3wmn.png)

The import created 221 Country nodes.

### 4.4 Importing Airport Nodes

The airports.csv file was used to create Airport nodes. The countryName property was retained on each Airport node to support matching, validation, and easier inspection during implementation.


### 4.5 Creating LOCATED_IN Relationships

After importing the Airport and Country nodes, LOCATED_IN relationships were created from each airport to its corresponding country or region.

```cypher
// Create LOCATED_IN relationships from Airport to Country.
LOAD CSV WITH HEADERS FROM 'file:///airports.csv' AS row
MATCH (a:Airport {airportId: toInteger(row.airportId)})
MATCH (c:Country {countryName: row.countryName})
MERGE (a)-[:LOCATED_IN]->(c);
```

![gh](https://cdn.jsdelivr.net/gh/extraterrestrial-being/markdown_images@main/1778607082000zg05sy.png)
Fig. 19. Import result for LOCATED_IN relationships.

The import created 2,795 LOCATED_IN relationships, which matches the number of airport nodes. This confirms that every airport was connected to one country or region.

### 4.6 Creating ROUTE Relationships

The routes.csv file was used to create directed ROUTE relationships between departure and arrival airports. Each row in routes.csv was imported as one ROUTE relationship to preserve the original route record count from the source dataset.

```cypher
// Import directed ROUTE relationships.
// Each source row becomes one ROUTE relationship to preserve route record counts.
LOAD CSV WITH HEADERS FROM 'file:///routes.csv' AS row
MATCH (dep:Airport {
    airportName: row.departureAirportName,
    city: row.departureAirportCity,
    countryName: row.departureCountryName
})
MATCH (arr:Airport {
    airportName: row.arrivalAirportName,
    city: row.arrivalAirportCity,
    countryName: row.arrivalCountryName
})
CREATE (dep)-[:ROUTE {
    routeRecordId: toInteger(row.routeRecordId),
    airlineName: row.airlineName,
    airlineCountry: row.airlineCountry,
    planeNames: row.planeNames,
    routeType: row.routeType
}]->(arr);
```

Fig. 20. Import result for directed ROUTE relationships.

The import created 57,301 ROUTE relationships. This matches the number of source rows used during ETL, confirming that each original route record was preserved as a relationship in Neo4j.

# 7. Graph Data Science Application: Airline Route Planning and Network Resilience

Graph Data Science can support airline route planning and network resilience by modelling airports as nodes and flight routes as relationships. This graph representation allows analysts to examine not only direct airport connections, but also multi-hop routes, alternative paths, and the minimum-cost network structure required to maintain connectivity. This is appropriate for aviation analysis because air transport networks are naturally graph-based systems, where airports represent nodes and flight services represent links. Previous studies on air traffic network resilience have also used network-based methods to evaluate connectivity, disruption response, and re-routing strategies [5]. 

One suitable algorithm is **Breadth First Search (BFS)**. BFS starts from a selected source node and explores all one-hop neighbours before moving to two-hop neighbours and further levels. In an airline network, BFS can be used to identify routes with the fewest number of flight segments between two airports when all routes are treated as having equal cost. For example, if a passenger needs to travel from Beijing to Perth and no direct flight is available, BFS can determine whether a one-stop or two-stop route exists. This is useful when the main optimisation objective is to minimise the number of transfers rather than distance, travel time, or ticket price. BFS is a standard graph traversal algorithm and is commonly used for shortest-path discovery in unweighted graphs [6]. 

A second suitable algorithm is **A* search**. A* is especially relevant for route planning because it combines the actual cost already travelled, g(n), with an estimated remaining cost, h(n), using the evaluation function f(n) = g(n) + h(n). In an airport network, the actual cost could represent flight distance, travel time, fuel cost, ticket price, or transfer penalty. The heuristic could be the straight-line geographical distance between the current airport and the destination airport. This makes A* suitable for finding efficient travel routes when airport coordinates and route weights are available. Compared with an unweighted traversal, A* can prioritise routes that are more likely to lead to a lower-cost solution. The original A* algorithm was introduced by Hart, Nilsson and Raphael as a heuristic method for determining minimum-cost paths [7].

A third useful algorithm is **Minimum Spanning Tree (MST)**. MST can be used to identify a low-cost backbone network that connects all selected airports in a region. If each route is assigned a weight such as distance, operating cost, or infrastructure cost, MST can find a set of connections that links all selected airports with the minimum total weight. This can support strategic planning tasks such as designing a minimum-cost regional connectivity network, identifying essential backbone links, or evaluating the basic structure required to keep an airport network connected. MST is a classical weighted graph problem, commonly solved by algorithms such as Kruskal’s and Prim’s algorithms [6]. 

These algorithms address different aspects of airline network analysis. BFS is suitable for fewest-hop route discovery, A* supports weighted route planning with geographical heuristics, and MST helps identify a low-cost connected backbone. Together, they show how Graph Data Science can support practical decision-making in aviation, including route recommendation, disruption response, and network design. This is consistent with transport network resilience research, where maintaining connectivity and adapting routes after disruptions are important objectives [5]. 

However, the usefulness of these algorithms depends on data quality and modelling choices. BFS ignores route weights, so a route with fewer transfers may still have a longer travel time or higher cost. A* requires meaningful route weights and reliable geographical information such as latitude and longitude. MST minimises total connection cost, but it does not directly consider passenger demand, airline competition, route capacity, or profitability. Therefore, these algorithms should be applied with clearly defined business objectives and carefully selected relationship weights.

**Best Algorithm for this scenario**

For the specific scenario of finding optimal travel routes between airports in the airline dataset — where route distances or travel costs can serve as relationship weights and airport coordinates can provide a geographical heuristic — A* search is the most suitable algorithm because it can efficiently identify minimum-cost paths without exhaustively exploring all possible routes.


## 8. Metadata in Graph Database Design and Cypher Query Development

Metadata is an important consideration in graph database design because it describes the structure of the graph rather than the individual business records. In Neo4j, metadata includes node labels, relationship types, property keys, constraints, indexes, and the observed structural patterns between labels and relationships. This is important because Neo4j uses the property graph model, where data is represented as nodes, relationships, labels and properties [1].

In graph database schema design, metadata helps define the conceptual structure of the domain. For this project, the main node labels are Airport and Country, and the main relationship types are ROUTE and LOCATED_IN. This metadata makes the intended semantics of the graph clear: airports are entities, countries or regions are entities, each airport is located in a country or region, and each ROUTE relationship represents a directed airline service record between two airports. Without this metadata, the graph would be difficult to interpret because developers would not know which labels and relationships represent business entities and which properties describe them.

Metadata also helps ensure schema consistency. For example, the project graph uses the property key countryName on Country nodes and also retains countryName on Airport nodes for import matching and validation. Similarly, ROUTE relationships use property keys such as airlineName, airlineCountry, planeNames, and routeType. Consistent metadata avoids errors caused by inconsistent names such as country, countryRegion, or CountryName. Neo4j’s naming guidance treats node labels, relationship types, property names, variables, indexes and constraints as important named schema elements, so consistent naming is essential for readable and reliable Cypher queries [2].

Another reason metadata is important is that it supports validation of the implemented graph against the intended design. The Arrows App design is a schema-level representation, but the real database must be checked after import. Metadata queries can confirm whether the expected node labels, relationship types, and property keys were created. In this project, database statistics showed Airport and Country labels, ROUTE and LOCATED_IN relationship types, and the expected relationship counts. This confirmed that the implemented graph matched the intended schema.

Metadata is also useful for identifying design limitations. For example, in this project Airline and AircraftType were not modelled as separate node labels. Instead, airline and aircraft information were stored as properties on ROUTE relationships. Metadata makes this design decision explicit. A developer inspecting the schema would understand that airline-centred traversal is not directly supported as a node-to-node pattern, while route-centred filtering and aggregation are supported through ROUTE properties. Therefore, metadata helps communicate both the strengths and limitations of the graph model.

Developers and data engineers can also use metadata to write more effective Cypher queries. Metadata tells the developer which labels and relationship types should be used in a MATCH pattern. For example, knowing that routes are represented by (:Airport)-[:ROUTE]->(:Airport) allows the developer to write path queries such as:

```cypher
MATCH path = (:Airport {airportName: "Beijing Capital International Airport"})
             -[:ROUTE*1..3]->
             (:Airport {airportName: "Perth International Airport"})
RETURN count(path);
```

This query depends directly on metadata: Airport is the node label, ROUTE is the relationship type, and airportName is the property key used to identify the start and end airports. This is consistent with Cypher’s role as Neo4j’s declarative query language for querying property graph databases [3].

Metadata also supports query efficiency. In Cypher, using specific labels and relationship types usually produces more selective and understandable queries than matching all nodes and relationships. For example, the query pattern:

```cypher
MATCH (dep:Airport)-[r:ROUTE]->(arr:Airport)
WHERE r.routeType = "international"
RETURN count(r);
```

is more precise than:

```cypher
MATCH (a)-[r]->(b)
WHERE r.routeType = "international"
RETURN count(r);
```

The first query uses metadata to restrict the search to Airport nodes and ROUTE relationships. This improves readability and can reduce unnecessary graph scanning. Metadata also helps identify which properties may benefit from constraints or indexes. In this project, uniqueness constraints were created on Country.countryName and Airport.airportId, because these properties are used repeatedly for node matching during import and validation. Neo4j constraints are designed to support data quality and integrity, while indexes can improve search performance when queries filter or match by specific labels, relationship types, and property names [8], [9].

Neo4j and APOC provide practical tools for inspecting graph metadata. For example, apoc.meta.schema() examines a graph and returns schema metadata as a map, while apoc.meta.stats() returns transactional database statistics such as node counts, relationship counts, labels, and relationship types [10], [11]. apoc.meta.graph() can also return a meta-graph representing labels and relationship types and how they are connected [12]. These tools help developers understand the structure of an existing graph, especially when the database is large or was created by another developer.

For example, apoc.meta.schema() can summarise labels, relationship types, and property keys. A data engineer could use it to confirm that ROUTE relationships contain airlineName, airlineCountry, planeNames, and routeType. apoc.meta.stats() can provide quick graph statistics, such as label counts and relationship type counts, which are useful for checking whether ETL produced the expected graph size. apoc.meta.graph() or db.schema.visualization() can show the schema pattern visually, helping developers confirm that Airport nodes connect to Country nodes through LOCATED_IN, and to other Airport nodes through ROUTE. Neo4j documentation also describes graph visualisation as a way to gain a “big picture” view of graph data and relationships [13].

Metadata is particularly valuable for debugging. If a query returns no results, metadata can help identify whether the issue is a wrong label, a wrong relationship direction, a missing property key, or inconsistent casing. For example, a query using [:route] instead of [:ROUTE] would fail because the actual relationship type in the graph is ROUTE. Similarly, querying r.airline_country would not work if the actual property key is airlineCountry. Inspecting metadata helps detect these problems quickly.

Overall, metadata is essential in graph database design because it defines and communicates the structure of the graph. It helps ensure that the implemented graph matches the intended schema, supports consistent ETL, and reveals the assumptions and limitations of the design. For Cypher development, metadata helps developers choose correct labels, relationship types, properties, constraints, and indexes. As a result, metadata improves both the effectiveness and efficiency of Cypher queries.
